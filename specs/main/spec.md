## 产品目标（用户体验）

用户在 **SocialWiz 内**完成与发文相关的全部主流程：

1. **登录的仅是 SocialWiz**（本应用账号），不要求用户去各平台完成 **开放平台应用创建、App Key/Secret、OAuth 授权回调** 等「开发者接入式」认证。
2. 用户选择已连接的 **目标平台账号**（连接方式见下节），编辑内容后点击 **发布**。
3. 系统在 **后台**通过 **非官方发布引擎**（浏览器自动化、对站内接口的受控复现等，按平台插件实现）将内容送达对应平台；用户感知为 **一次点击，与在平台内直接发布等价**（成功/失败以站内状态与文案反馈）。

> **措辞说明**：「避开平台接入相关的认证」指 **不依赖各平台开放平台的 OAuth/应用凭证主路径**；用户仍可能需要 **在受控流程中登录目标平台网页账号**（例如 Playwright 拉起浏览器扫码/输密码），以取得 **网页会话**，这与「注册开发者应用 + 拿 token」是不同概念。

## 非官方发布引擎（主路径）

**定义**：按平台的 **`PlatformPublishPlugin`**（命名随实现），在拥有有效 **网页会话**（或等价登录态）的前提下，执行发帖、传图、传视频等。实现手段对上层 **抽象为统一接口**，可包括：

- **浏览器自动化**（默认、可维护性优先）：Playwright 等加载 storage state，操作发布页。
- **HTTP 复现 / 逆向**（可选、按平台封装）：在插件内维护 Cookie、CSRF、签名与端点版本；**不**要求用户配置开放平台参数。

**扩展**：新增平台 = 新插件 + 会话采集流程 + manifest（内容类型、是否支持无头、推荐 transport）。

**与 `003` / OAuth 代码的关系**：现有 `WeiboAdapter`、OAuth 路由可作为 **兼容或迁移期**保留；**产品默认 UI 与主发布链路不引导用户走开放平台绑定**。若某平台同时存在两种实现，由服务层按账号连接类型或功能开关路由。

## 用户主流程（相对旧版澄清的替换描述）

以下取代原先「跳转微博开放平台授权」作为主流程叙述：

1. 用户在 SocialWiz 内发起 **连接某平台账号**（文案避免强调「OAuth」「应用密钥」）。
2. 系统启动 **会话采集**（如子进程 + 浏览器）：用户在 **目标平台网页**完成登录；成功后持久化 **按用户隔离的会话制品**（见 `data-model.md`）。
3. 回到 SocialWiz，列表展示已连接账号；用户进入 **发布**，选择账号并提交。
4. **发布请求**进入统一队列/任务：后台引擎加载对应插件与会话，执行发布；前端可轮询或使用统一任务状态 API 展示进度与结果。
5. **数据/统计**（迭代）：同样优先通过 **网页会话可访问的能力** 或插件内封装实现；不假设开放平台只读 API 可用。

---

## 浏览器会话 Uploader（与引擎的关系）

此前单独描述的「浏览器会话 Uploader」并入 **非官方发布引擎** 的默认 transport；**bind / compose** 语义保留为：**bind** = 采集会话，**compose/publish** = 执行发帖。

---

## 范围摘要

在 SocialWiz 中建立 **以非官方发布引擎为主路径** 的多平台发布能力：**无开放平台认证主流程**、**站内一键发布** 的端到端体验；微博为首个完整参考实现，架构可横向扩展。符合 Constitution 的前提下，业务逻辑集中于服务层，平台差异收敛于插件与 manifest。

## 当前实现基线（截至规划日）

- 已有：微博 Playwright `playwright-bind`、`compose-runner`（HTTP 复现发博）、`WeiboAdapter` 与 OAuth 相关路由、账户页与统一发布接口。
- **待产品化**：统一发布入口对「会话型账号」走引擎而非开放平台 `publish`；弱化或隐藏 OAuth 主路径；可选引入 **HTTP 复现型** 平台插件接口；生产环境 **执行面**（worker、会话存储）需单独设计。
