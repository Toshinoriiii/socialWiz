## User Scenarios & Testing *(mandatory)*

### User Story 1 - 运营同学快速在左右布局中完成日常操作 (Priority: P1)

作为已登录的运营用户，我进入重构后的 dashboard 页面时，左侧看到清晰分组的主导航（主页、数据分析、数据概览、发布管理、账号管理、平台管理、设置），右侧为当前模块的主要工作区域；我希望可以在不离开当前视线区域的情况下完成查看数据、创建内容等高频操作。

**Why this priority**: 这是 dashboard 的核心价值，关系到日常使用效率和整体交互感受，是本次迭代最关键的目标。

**Independent Test**: 仅实现左右布局和导航切换，也能通过可用性测试（例如 5 名运营用户在 3 分钟内完成指定导航跳转和数据浏览任务），形成可演示的 MVP。

**Acceptance Scenarios**:

1. **Given** 用户已登录并进入 dashboard，**When** 点击左侧任一主导航标签（如「数据分析」），**Then** 右侧工作区无整页刷新、平滑切换到对应模块内容且导航高亮状态正确。
2. **Given** 用户在右侧工作区滚动浏览，**When** 页面高度较大时，**Then** 左侧导航在可视区域内保持固定，保证随时可切换模块。

---

### User Story 2 - 使用黑白极简风快速理解当前状态 (Priority: P2)

作为运营用户，我希望 dashboard 的整体视觉是黑白为主、信息层级清晰，重要信息通过有限的强调色、shadcn 组件的结构化样式来体现，而不是大量色块和装饰，从而在长时间工作下依然保持舒适、易读。

**Why this priority**: 视觉改造直接影响整体品牌感和使用舒适度，是本次迭代的主要设计目标之一。

**Independent Test**: 即便暂时不接入所有真实业务数据，仅加载占位数据和基础组件，也可以评估信息层级是否清晰（通过可用性测试和设计走查）。

**Acceptance Scenarios**:

1. **Given** 用户首次进入新 dashboard，**When** 浏览左侧导航与右侧主要内容区域，**Then** 页面主体颜色以黑、白、灰为主，强调色数量有限且不干扰阅读。
2. **Given** 用户查看主要数据卡片、操作按钮，**When** 快速扫视页面 3 秒内，**Then** 能准确指出当前页面的主操作（如「创建文章」）与主要数据区域。

---

### User Story 3 - 高效访问创作与数据模块 (Priority: P3)

作为经常在平台内创建内容并查看表现的运营用户，我希望可以通过左侧导航和右侧顶部的快捷入口，快速完成「创建文章 / 创建图文 / 创建视频」以及查看对应的发布表现和数据概览，而不需要在多个页面之间频繁跳转。

**Why this priority**: 创作入口和数据表现是当前业务最常用的操作路径，优化之后可以明显降低路径深度和时间成本。

**Independent Test**: 在仅实现创作相关模块和数据概览的情况下，测试用户能够在 3 步以内完成从进入 dashboard 到打开某个创作表单页面的操作。

**Acceptance Scenarios**:

1. **Given** 用户在 dashboard 首页，**When** 点击右侧工作区内的「创建文章 / 创建图文 / 创建视频」入口，**Then** 系统在同一布局体系下打开对应的创作表单或跳转到相应创作页面。
2. **Given** 用户在「数据概览」模块，**When** 点击某个数据卡片（例如某篇作品的阅读量卡片），**Then** 右侧能够进入更详细的明细或筛选页面，而左侧导航仍保持当前模块高亮。

---

### Edge Cases

- 当浏览器宽度过窄（例如小于一定断点）时，左侧导航在一定宽度以下自动折叠为仅图标栏，并提供显式按钮手动展开 / 收起；在更窄宽度下导航以抽屉形式出现，点击图标或菜单按钮后侧滑显示。
- 当当前用户权限不足以访问某些模块（如「平台管理」）时，左侧导航和右侧工作区应如何表现（隐藏菜单，还是禁用并展示提示）。
- 当 dashboard 数据接口返回为空或异常时，右侧数据卡片和列表的空状态与错误状态展示方式。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须将 dashboard 页面整体改为左右结构布局，左侧为主导航栏，右侧为工作区内容区域；布局在常规桌面分辨率下保持稳定，不出现水平滚动条。
- **FR-002**: 左侧导航必须包含以下主标签：主页、数据分析、数据概览、发布管理、账号管理、平台管理、设置，并支持根据当前路由或当前模块高亮对应标签。
- **FR-003**: 右侧工作区必须根据左侧导航选中项展示对应模块的页面内容，切换导航时应保持页面级别路由一致（可通过浏览器前进/后退返回到对应模块）。
- **FR-004**: 导航与工作区均应基于本项目现有的 shadcn 组件体系（如 `Sidebar`、`Button`、`Card`、`Tabs` 等）实现，不再使用自定义非规范组件库样式。
- **FR-005**: Dashboard 整体视觉必须遵循黑白极简风设计，主要背景为白色或浅灰，文本为黑色或深灰，边框与分割线为低饱和灰色；图表主数据线和关键状态标签可以使用品牌主色及少量辅助色进行区分，但整体色彩数量应控制在 2–3 种，保持黑白极简基调不被破坏。
- **FR-006**: 在「主页」中，需提供若干核心信息卡片，至少包括「作品数据概览」「账号数据概览」「发布数据概览」「互动数据概览」四大块，并按照信息优先级从左到右、从上到下排列，优先保证这四类信息在首屏即可看到；字段选择在示例图基础上做合理预设，后续可根据业务反馈微调。
- **FR-007**: 在「主页」中，应提供快速创作入口区域，至少包含「创建文章」「创建图文」「创建视频」三个操作按钮，点击后进入各自的创作流程页面。
- **FR-008**: 所有按钮、链接等可交互元素应具备明显的 hover / active 状态，且与黑白极简风保持一致（例如：使用阴影、边框变化或灰度变化，而不是高饱和底色块）。
- **FR-009**: 左侧导航在垂直方向上支持滚动，当菜单项较多超过视口高度时，用户仍可访问全部菜单，同时当前选中项在滚动时保持可见。
- **FR-010**: Dashboard 页面在加载时应有统一的 skeleton 骨架屏或 loading 状态，避免出现大面积空白或布局跳动。

### Key Entities *(include if feature involves data)*

- **DashboardSidebarItem**: 表示左侧导航的一个主模块项，包含属性：标识符（如 `home`、`analytics`）、显示文案、图标类型、是否可见（基于权限）、对应路由路径。
- **DashboardSectionCard**: 表示右侧工作区中的一个信息卡片或入口卡片，包含属性：标题、副标题/说明、主要数据值或操作按钮、数据来源标识、点击行为（跳转或弹层）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在可用性测试中，至少 80% 的运营用户可以在 3 分钟内完成指定的三项任务（例如切换到「数据分析」、找到「创建文章」入口、查看作品数据概览）且无需外部指引。
- **SC-002**: 页面在常见桌面分辨率（例如 1440×900 及以上）下首次加载完成、主布局和导航可见的时间不超过 3 秒（以本地测试为准）。
- **SC-003**: 在设计评审和体验走查中，相关干系人（产品、设计、核心运营）对「布局是否清晰」「信息是否易读」两个维度的满意度评分平均不低于 4/5。
- **SC-004**: 上线后一个迭代周期内，与「导航找不到入口」「页面信息太杂/太花」相关的反馈或内部投诉数量相较于当前版本减少 50% 以上。
